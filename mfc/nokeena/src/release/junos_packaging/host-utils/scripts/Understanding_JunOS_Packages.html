Understanding packages

This (incomplete) doc provides a description of how packages work in JUNOS.

JUNOS packages, are just like those found on BSD, SunOS and similar operating systems. They are created by pkg_create installed via pkg_add and removed by pkg_delete.

What is a package

In the most generic sense a package is simply a gzipped tar file (.tgz) that contains the following files:

+COMMENT
A one line description of the package. In JUNOS this is displayed by show version.
+DESC
A more complete description of the package.
+CONTENTS

This is the list of files in the package as well as MD5 hashes and various directives - it is generated from a PLIST by pkg_create.

The key difference between PLIST and +CONTENTS that the later has a comment after each file entry containing an MD5 hash. This is used by both pkg_add and pkg_delete to verify that all is well.

files
The actual files that comprise the content. For most JUNOS packages this is a single file which is an ISO file-system image, plus ancillary files to verify its integrity. See verify_file below.

and may also contain the following optional files:

+REQUIRE
An optional script (most JUNOS packages have this), which checks if the package is compatible with the system and existing softwarre.
+INSTALL

An optional script (all JUNOS packages have this), which controls the pre and post install activities.

Normally pkg_add does the actual copying of files to their destination. This is supressed in JUNOS packages. See below.

+DEINSTALL
An optional script (many JUNOS packages do not have this), which controls the pre and post actions when removing a package.

Creating packages

This is performed duing the build using pkg_create. Which does a little bit more than creating a tar file based on the contents of the given PLIST

PLIST

The Packing LIST (PLIST) controls what goes into the package. The majority of JUNOS packages use the following simple PLIST:

@comment Copyright (c) 1996-2004, Juniper Networks, Inc.
@comment All rights reserved.
@comment
@cwd /
@srcdir %SRCDIR%
@option preserve
@name %PKGNAME%
@owner root
@group wheel

@ignore_inst
packages/%PKGFILE%
@ignore_inst
packages/%PKGFILE%.sha1
@ignore_inst
packages/%PKGFILE%.sig
@ignore_inst
packages/%PKGFILE%.certs
@srcdir %OBJDIR%
@ignore_inst
%PKGNAME%.symlinks

Lines in the PLIST which are not empty and do not start with @ name files to be included in the package (and hence by default installed).

The @ directives are documented in pkg_create(1), we will mention only the interesting ones below.

The various %*% works are subsituted for values supplied by the makefiles. For example when building the jkernel64 package:

PKGFILE=jkernel64-${RELEASE}
PKGNAME=jkernel
SRCDIR=${SB}/obj-amd64-mtx/mtx/release/rootdir
OBJDIR=${SB}/obj-amd64-mtx/mtx/package-scripts/jkernel64

@name

The @name field controls what the package will be registered as. This is important to understand, since pkg_add will fail if the package name is already registered as installed.

The keen observer will note in the example above that PKGNAME (value of @name) is set to jkernel rather than jkernel64. This is a deliberate abuse of the package system to ensure that the packages jkernel-10.3 and jkernel64-10.3 are treated as mutually exclusive.

It also ensures that most of the system can be blissfully ingnorant of the distinction between the two packages.

@ignore_inst

This tells pkg_add to not install the file. JUNOS packages handle that via the +INSTALL script to better control the filesystem layout.

@comment

Simply a comment, but if you look at the +CONTENTS files under /etc/db/pkg/* you will see that each file entry is followed by and MD5 digest:

+REQUIRE
@comment MD5:da153d21bc7bfc17c0807ecaaf8c2f94

we also use @comment to mask optional content, which is enabled in some packages by stripping the @comment token, turning the line into a file entry.

@srcdir

Indicates where to grab the actual files from.

Makefiles

There are a lot of makefiles involved in generating packages. The ones of specific interest here are found in junos/package-scripts/ in particular junos/package-scripts/Makefile.inc which contains all the logic for dealing with the %*% substitutions above.

The makefiles for building the ISO images are beyond the scope of this document.

Installing packages

While running pkg_add directly is not supported, that is what eventually happens when a user does request system software add.

Note: This section covers just the mechanics.

-signed packages

As noted above, pkg_add will fail to add jroute if the jroute package is already installed. This means that one must pkg_delete the old package before adding the new. That can (and has) resulted in disaster, if something goes wrong.

The JUNOS -signed.tgz packages were introduced to solve that problem among others. These are just wrapper packages which all use the same PLIST:

@comment Copyright (c) 2001, 2004, 2007, Juniper Networks, Inc.
@comment All rights reserved.
@comment
@name %PACKAGE%-signed
@srcdir %SRCDIR%
@option no-record

@ignore_inst
%PACKAGE%.tgz
@ignore_inst
%PACKAGE%.tgz.md5
@ignore_inst
%PACKAGE%.tgz.sha1
@ignore_inst
%PACKAGE%.tgz.sig

@srcdir %CERTS_OBJDIR%
@ignore_inst
certs.pem

These packages do not themselves have a +REQUIRE script. All of the logic is in +INSTALL which:

  1. verifies the integrity of the contained package (see verify_file) The actual verification methods used have evolved over the years.
  2. extracts +REQUIRE from the contained package and runs it. If successful we presume that adding the contained package will succeed.
  3. pkg_delete the old package and pkg_add the new.

The reason this can work is that the @name_ field of the -signed package is not the same as the target package.

verify_file

JUNOS packages are just one place where the following model applies.

Over the years JUNOS has evolved from using MD5 (.md5), to SHA1 (.sha1) and more recently RSA+SHA1 digital signatures (.sig). In the near future we will need to add support for something like ECDSA+SHA256 signatures (.esig?).

The functions in JUNOS which verify files, look for the strongest method available - that they understand.

Depiste the fact that JUNOS has not made use of the .md5 files found in many places in JUNOS, they are retained to facilitate upgrading from ancient JUNOS versions which do not grok .sha1 or later methods.

Thus for a given file we may see:

file            the actual file
file.md5        contains an MD5 digest of file
file.sha1       contains a SHA1 digest of file
file.sig        contains a SHA1 digest of file encrypted with RSA
file.certs      (or certs.pem) the PEM encoded X.509
                certificate chain needed to verify file.sig

hashes

Whether it is MD5 or SHA1, the verification method is the same, compute the digest and compare to the saved copy.

signatures

See the Signed packages of the JUNOS-FIPS spec for more details.

Basically verification relies on computing a digest (SHA1) and comparing to an stored version after decrypting it using the relevant public key (RSA).
